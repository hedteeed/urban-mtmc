Section 01 · packages/mantis_data

The data engine — half of all CV is this

Nobody puts dataset plumbing in a tutorial, yet it decides more of your final accuracy than the model does. Mantis ingests nine public datasets — drone traffic (VisDrone, UAVDT, AU-AIR), overhead vehicles (VEDAI), satellite scenes (DIOR, DOTA), maritime (SeaShips, SMD, SeaDronesSee) — each with a completely incompatible label format: CSV rows, Pascal VOC XML, one giant JSON, MATLAB files, 8-coordinate rotated boxes. The architecture that tames them is worth memorizing.

The adapter + taxonomy pattern

Every dataset gets an adapter (~150 lines) that knows only its own format quirks. Every decision — which native class maps to which unified class — lives in one declarative YAML file, configs/classes_unified.yaml, which flattens everything into 9 classes (person is id 0; then car, truck, bus, two_wheeler, small_boat, cargo_vessel, passenger_vessel, aircraft). VisDrone's "pedestrian" and "people", SeaDronesSee's "swimmer" — all fold into person. A native class with no mapping is silently dropped. The file's header even forbids putting mapping decisions in adapter code.

Lesson · label-space harmonization

When you combine datasets (and you will — CrowdHuman + MOT20 + your own mall footage), the merge decisions are policy, not code. Keeping them in one reviewable YAML means you can audit "does a child on a scooter count as person?" without reading Python. Copy this pattern verbatim.

The output format is COCO JSON — the lingua franca of detection: an images list, an annotations list with bbox = [x, y, width, height] in pixels, and a categories list. Mantis extends it with provenance keys (source, source_id, original class name, license, commercial_use) that standard tools ignore — lineage rides inside the file forever. One subtle move: every per-dataset file always declares all 9 categories even if it only populates two, so category ids can never drift between files when merging.

Deduplication — why video makes dangerous training data

Three of the datasets are videos sampled into frames. Consecutive frames are near-identical. If you shuffle and split them randomly, near-copies of the same scene land in both train and eval — the model gets graded on data it memorized, and your accuracy numbers become fiction. This is called leakage, and it is the single most common way beginners fool themselves.

The defense here is a perceptual hash (dedup.py): shrink every image to 9×8 grayscale, compare each pixel to its right neighbor, produce a 64-bit fingerprint of the image's gradient structure. Two frames whose fingerprints differ by ≤5 bits (Hamming distance) are declared duplicates and one is dropped — before splitting. It's invariant to re-encoding, resizing, and brightness shifts: exactly the differences between "the same frame stored twice."

Verified gap · residual temporal leakage

The split (union.py) is per-image, stratified per-source. Frames from the same video sequence that differ by more than 5 hash bits (camera panned, cars moved) survive dedup and can still straddle the train/eval boundary. For detection this inflates scores mildly. For your product it's fatal: tracking and Re-ID data must be split per-sequence — the same person seconds apart must never appear in both train and eval. Demand this in every dataset spec you sign off on.

Two more moves worth knowing

  • Stratified split with a floor. Each source contributes 10% of its own images to eval, minimum 1 — otherwise UAVDT's ~80k frames would drown VEDAI's 1,250 and you'd lose all evaluation signal for small domains. Seed is fixed at 42; identical inputs give byte-identical splits. Reproducibility is a feature you specify, not luck.
  • Tiling giant images. DOTA scenes reach ~13,000px. Downscaling would erase the tiny vehicles that are the whole point, so the adapter cuts 1024×1024 tiles with 200px overlap; a ground-truth box joins a tile only if ≥50% of its area is inside. This is training-time SAHI — the same slice-and-merge idea your guide describes for inference.

Also verified: the adapters uniformly discard occlusion/truncation flags the datasets provide, keep zero pure-background images (a known cause of false positives), and never apply UAVDT's "ignore regions" — so unlabeled cars there teach the detector that visible cars are background. Each of these is a deliberate-or-lazy trade you now know to ask about.

Section 02 · packages/mantis_detector

The detector — YOLOX-S, end to end

The model is YOLOX-S — an anchor-free, one-stage detector from the YOLO family — consumed through the Apache-2.0 pixeltable-yolox library. "S" means small: depth multiplier 0.33, width 0.50. The repo's own code is deliberately thin: a config, a dataset bridge, MLflow wiring, an evaluator, an exporter. The network, losses, and training loop come from the library untouched. That's the professional default — you don't write detectors, you configure them.

Why YOLOX and not the famous Ultralytics YOLOv8/v11? Licensing. Ultralytics is AGPL-3.0 — importing it into a proprietary product legally obliges you to open-source the product. LICENSES.md names this trap explicitly and even warns that Ultralytics' dataset YAML files carry AGPL headers. This single fact will matter to your mall product on day one, because every tutorial you'll find uses Ultralytics.

The four decisions in the config

config.py changes exactly four things from stock YOLOX-S, and each is a lesson:

  1. 9 classes — the head's classification branch matches the taxonomy.
  2. 1280×1280 input instead of the 640 default. Aerial targets are tiny; resolution is the cheapest small-object fix (your guide's Part 2). Cost: ~4× compute, absorbed by dropping batch size to 8. Indoors, people are large — you'll run 640–960 and bank that compute for more cameras.
  3. Eval every 50 epochs instead of 10 — because in-training multi-GPU evaluation crashed an 8×A100 run twice (an NCCL timeout, dated in a comment). Reading war stories in comments is how you learn what actually breaks.
  4. Everything else inherited — 300 epochs, mosaic + mixup augmentation, cosine LR, EMA — with a comment: "intentionally untouched so this is a baseline." Establish a clean reference before tuning. Always.
Lesson · the augmentations that matter

Mosaic stitches 4 random training images into one, jittering scale 0.1–2×; mixup blends image pairs. Both force the model to see objects at unusual scales and contexts — powerful for small objects, and standard in every modern YOLO recipe. The last 15 epochs run augmentation-free so the model finishes on clean, realistic imagery. For your mall model, the analog is random erasing (from your deep-dive guide) — synthetic occlusion, because occlusion is the indoor enemy.

One genuinely surprising verified fact: training starts from random weights. There is no COCO-pretrained checkpoint anywhere — 300 from-scratch epochs is the plan. Defensible for a specialized 9-class aerial domain; for your people detector, you will do the opposite (start from pretrained weights, fine-tune on people data) and converge in a fraction of the time.

Inference: the five steps every frame takes

inference.py is the cleanest 200-line summary of real detector plumbing you'll find:

  1. Letterbox — resize preserving aspect ratio into a 1280×1280 canvas, pad the remainder with gray (value 114), keep the scale ratio. YOLOX quirk: image anchored top-left, pixels stay BGR 0–255, no normalization.
  2. Forward pass — the model emits ~thousands of candidate rows: center-x, center-y, w, h, objectness, 9 class scores.
  3. Score & threshold — score = objectness × best class score; drop below conf (0.30 in the tools).
  4. Un-letterbox — divide coordinates by the saved ratio to get original-frame pixels.
  5. Class-aware NMS at IoU 0.65 — duplicate boxes of the same class merge; a person box never suppresses an overlapping car box.

Those two numbers — conf and NMS IoU — are your guide's "two knobs you tune forever," and the repo quietly proves the point: the CLI default is 0.30, but the Studio web form ships 0.6. Someone tuned the demo for a clean map over recall and never reconciled the defaults. You've now caught your first real config drift.

Evaluation and export

Eval uses pycocotools COCO mAP (AP@50:95, AP50, AP75) — but the smart part is per-source scoring: the same detections re-scored per dataset domain, because an aggregate number hides "great on land, blind at sea." At eval time the confidence threshold drops to 0.01 — mAP integrates the whole precision-recall curve, so eval wants everything; only display wants 0.30+. Export goes to ONNX (portable model graph) with a mandatory parity check: run the same input through PyTorch and ONNX Runtime, require answers to match within tolerance, exit nonzero otherwise. No TensorRT, no INT8 quantization anywhere — the deployment story stops at ONNX. Your guide's Part 6 optimization chain remains yours to build.

Section 03 · packages/mantis_reencoder

Time & telemetry — the hidden backbone

This is the subsystem with the least YouTube-tutorial glamour and the most transferable wisdom. A drone's video is useless for mapping without knowing, per frame, where the camera was and where it pointed. Militaries standardized that: KLV (key-length-value binary packets) per MISB ST 0601 (which fields mean what), muxed into the video file per STANAG-4609. Mantis implements a 26-tag encoder/decoder from scratch: platform lat/lon/altitude, heading, gimbal angles, field of view, slant range, frame center — each value squeezed into 2–4 bytes by mapping its physical range onto an integer range (latitude resolution works out to ~5mm).

You will never use KLV in a mall. You will use its lessons daily:

Lesson 1 · streams are joined by time, never by index

Telemetry arrives asynchronously, at a different rate than video. So every packet carries its own microsecond timestamp (Tag 2), every video frame gets a computed timeline position, and the join is nearest-timestamp within a tolerance (±2 frame periods here) via binary search. Outside tolerance → the frame honestly reports "no pose" instead of using a stale one. Your mall system is nothing but this problem multiplied: 20 cameras, each with its own clock, fused by time. Clock sync (NTP/PTP) plus this exact nearest-with-tolerance join is the backbone of cross-camera identity — your deep-dive guide's section C7, here as working code.

Lesson 2 · never trust a codec you only round-tripped

A systematic scaling bug encodes and decodes consistently — your own tests pass while every value is wrong. So test_klv_oracle.py decodes a real 2009 drone sample through an independent third-party parser and requires tag-for-tag agreement. This is oracle testing. Apply it to anything you build that has a spec: validate against an independent implementation and real captured data, not your own mirror image.

Lesson 3 · synthetic data buys you the pipeline before reality arrives

The real test flight looked 0.37° above horizontal — a useless grazing view. So synth.py generates a fake drone orbiting a target at a steep 40°, with every field derived so the geometry is self-consistent by construction: the image center provably reprojects onto the target. The whole pipeline and dashboard were built and regression-tested against synthetic telemetry before real footage existed. Your version: simulate people walking a floor plan through simulated camera views, and build your entire tracker + dashboard before a single camera is mounted.

Also here: the muxer copies H.264 video bit-for-bit (never re-encode what you only need to re-wrap), and timestamps ride in the data rather than trusting the container's transport timing — because the container demonstrably mangles it (GStreamer ignores the requested synchronous-KLV mode; the code documents the deviation and works around it by construction).

Section 04 · geolocate.py + dji.py

Geometry — pixels onto the world

This is mantis's answer to the question your whole product hinges on: a detector found something at pixel (px, py) — where is it in the world? The drone answer, 77 lines in geolocate.py, verified line by line:

  1. Meters-per-pixel at the aim point: GSD = 2 · slant_range · tan(HFOV/2) ÷ image_width. This one formula is the camera intrinsics — focal length derived from field of view, assuming a centered, undistorted lens.
  2. Look-down angle (depression) = asin(altitude ÷ slant_range) — pure flat-ground trigonometry.
  3. Pixel offsets from image center: sideways offset = du × GSD; forward offset = −dv × GSD ÷ sin(depression). That division is foreshortening: the shallower you look, the more ground each image row covers. A floor of sin ≥ 0.15 stops near-horizon views from projecting to infinity.
  4. Rotate by look azimuth into north/east meters, convert to degrees (111,320 m per degree of latitude, cos-scaled longitude). Done.

The crucial subtlety, confirmed in source: this is not a true ray-cast and not a homography. It's a linearization — exact at the image center (anchored to the telemetry's reported frame-center coordinate), first-order approximate everywhere else, with error growing toward the frame edges. Camera roll is ignored entirely (DJI gimbals stabilize it to ~0). The docstring is honest: "real-ish, not survey-grade: ignores Earth curvature, terrain, and lens distortion."

Lesson · why your problem is easier — and where it bites the same

Mantis approximates because its camera moves every frame and it only has noisy GPS + gimbal telemetry. Your mall cameras are bolted down. A fixed camera over a flat floor admits an exact 3×3 homography — calibrate once by clicking 4+ floor points, correct for every pixel forever. Do not copy this math; use cv2.findHomography. But every failure mode transfers directly:

• "Flat ground" breaking on terrain = "flat floor" breaking on stairs, escalators, mezzanines — you'll need per-region planes.
• The sin(depression) blow-up = a CCTV camera staring down a long corridor at a shallow angle localizes distant people terribly — mounting angle beats resolution.
• Projecting a pixel assumes that pixel is on the floor — hence bottom-center of the box (the feet), and occluded feet corrupt position exactly like terrain corrupts it here.
• Spec-sheet FOV (the DJI's 73.7° comes from a lens spec, not measurement) = the quick hack; per-camera checkerboard calibration with a distortion model is the real version — mandatory for the fisheye lenses malls love.

The DJI side (dji.py) holds two more habits worth stealing. First, units-and-conventions quarantine: every uncertain assumption about DJI's metadata (degrees or radians? absolute or relative yaw? altitude above sea or above takeoff?) is isolated in one frozen options object, so reconciling against a real clip is a one-line fix. Choosing altitude-above-takeoff over sea-level altitude matters enormously — the wrong one inflates every distance by ~20×. Second, forward/inverse round-trip testing: the tests assert that deriving the aim point from a pose, then projecting the image center back, lands exactly on the aim point. Your version: reproject known floor markers through each camera's homography nightly and alert if closure drifts — a free "did someone bump the camera?" alarm.

Section 05 · server/ + web/

The dashboard — anatomy of the product surface

The demo surface is a FastAPI backend plus a React dashboard with exactly three runtime dependencies (react, react-dom, leaflet — even the map wrapper library was rejected over its license). The layout is already your guard console: video panel with detection overlays, live map with moving markers, tracked-objects table, telemetry strip. Study four patterns and one hard truth.

Pattern 1 — the frozen manifest contract

The pipeline emits one JSON file — per frame: timestamp, camera pose, detections with pixel box, class, score, and world coordinates — governed by a versioned schema document (docs/manifest_schema.md) with explicit invariants the UI may rely on, e.g. "no pose ⟹ positions null but boxes still present." The UI performs no geometry and never branches on data source. This producer/consumer discipline is exactly how your tracking backend should talk to your dashboard: schema first, invariants written down, UI as pure consumer.

Pattern 2 — one clock rules everything

A single animation-frame loop reads the video's current time, converts it to the telemetry timeline, binary-searches the nearest frame entry — and every panel renders from that one frame index. Video, boxes, map, telemetry can never drift, including while scrubbing. Multi-panel dashboards die of desync; this is the cure, and it generalizes to multi-camera scrubbing.

Pattern 3 — the marker pool

Map dots are kept in a pool keyed by track ID; each frame the code moves existing markers rather than recreating them, deleting only dead tracks. Result: dots glide instead of flickering. Same pattern, floor plan instead of map — and note Leaflet has a CRS.Simple mode that renders an indoor floor-plan image with meter coordinates instead of world tiles. Your mapping library is already chosen.

Pattern 4 — human-in-the-loop calibration

The Calibrate panel is four sliders — FOV scale, yaw offset, north/east nudge — that warp the already-projected points live until the dots sit on the road, then save the correction as data into the manifest. No pipeline re-run, instant feedback. This is precisely your per-camera homography-touch-up tool: an operator nudges until dots sit in the right corridor, saves, done. The most directly reusable UX idea in the repo.

The browser tracker — a miniature of your real problem

Because the detector emits independent per-frame detections with no IDs, the dashboard builds tracks itself (tracks.ts): greedy nearest-neighbor matching, same class only, gated at 15 meters; unmatched detections start new tracks; tracks unseen for 20 frames die; positions and speeds smoothed by exponential moving average. Read it as your guide's Part 3 in embryo — birth, association gate, death, smoothing — minus everything that makes tracking trustworthy: no motion prediction (Kalman), no optimal assignment (Hungarian), no low-confidence recovery (ByteTrack's trick), no appearance features. Two same-class objects passing within 15m will swap IDs. It's honest about being cosmetic. Your product moves this server-side and does it properly.

The hard truth · nothing here is live

Despite the "LIVE" badge in the HUD (verified: it just means "video is playing"), the entire product is post-hoc: upload a file → a single-worker queue runs the pipeline (one job at a time, deliberately, so the GPU can't be double-booked) → browser polls every 1.5s → plays back a precomputed manifest. No RTSP ingest, no streaming inference, no push updates, no auth on any endpoint. The inference loop itself is every-frame, batch-of-one, single-stream. Your guide's Part 6 / Section C — frame sampling, cross-camera batching, decode limits, backpressure — is entirely unbuilt here. When you scope your product, the live layer is a first-class build item, not an upgrade.

Section 06 · CI, licensing, provenance

The discipline layer — what separates a repo from a product

Mantis's most sophisticated subsystem contains no computer vision at all. It's the machinery that governs what's allowed to enter the codebase and leave it — and for a company selling proprietary software built on open-source ML, that machinery is existential.

Three layers of license enforcement

  • At commit: a pre-commit hook scans every Python dependency's declared license and refuses GPL/AGPL. You literally cannot commit a forbidden dependency.
  • At merge: CI re-runs the same gate, plus an equivalent gate over all npm packages (which would mechanically reject react-leaflet's "ethical source" license).
  • Weekly: a scheduled scancode job scans the source files themselves for embedded GPL/AGPL headers — catching code copy-pasted from, say, an Ultralytics repo, which no dependency scan can see.

Exceptions require written justification in the code. The policy doc, the scripts, and CI all agree. For your product — where the tempting default stack (Ultralytics) is AGPL and the classic datasets are non-commercial — copy this apparatus wholesale on day one.

Provenance: models inherit their data's law

RESEARCH_ONLY.md encodes an idea most teams learn painfully: weights trained on non-commercial data are non-commercial artifacts. All nine mantis datasets are commercial-use false or unclear, so Month-1 weights may never ship, may not even be fine-tuned into a product model. The policy defines MLflow tags (research_full / commercial_clean / synthetic_only) and claims the trainer refuses to launch without a consistent tag.

Verified gap · the enforcement doesn't exist

I grepped the entire repo: dataset_set appears only in the policy document. No code sets it, checks it, or blocks training without it. The document even forbids "removing or weakening" an enforcement that was never written. The policy is real; the mechanism is aspirational. Two lessons: (1) when a doc says "the system enforces X," verify in code — you just watched it fail in a professional repo; (2) for your product this is the pattern to implement for real, because your world has the same trap: Market-1501, MSMT17, MOT17/20, CrowdHuman are all research-grade licenses, and under GDPR your training data provenance is a legal question, not just a contractual one.

Smaller habits worth stealing

  • One lockfile, frozen everywhere — dev, CI, and Docker all resolve the identical environment; drift fails the build.
  • Two Dockerfiles — a fat CUDA training image and a slim, non-root runtime image (your training rig vs your per-site edge box), including a documented trap: syncing dependencies inside the GPU container would silently replace GPU PyTorch with the CPU build "and training would never use the card."
  • Strictness where bugs are silent — the type checker is permissive globally but maximally strict on the one package (the binary telemetry codec) where a type confusion corrupts data quietly. In your product, that package is the cross-camera fusion engine.
  • 1MB commit cap — model weights and datasets mechanically cannot enter git; they flow through DVC (data version control). Though verified: the DVC remote is a path in one developer's home directory and no data is actually tracked yet — scaffolding, not practice.

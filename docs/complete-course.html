One second in the life of the finished system

Hold this narrative — every section below zooms into one step of it. A camera in the north corridor delivers 25 frames this second; your system decodes them all but runs the expensive detector on only 5. The detector looks at each sampled frame and outputs rectangles: "person, 91% confident, here." The tracker — which has been following everyone in this corridor — predicts where each known person should be by now (Kalman filter), matches predictions to the new boxes (Hungarian algorithm), and keeps IDs stable: the woman in the red coat is still local track 12. For every healthy track, the Re-ID embedder looks at the person's cropped image and distills it into 512 numbers — her appearance signature — which is averaged into the track's running template. The geometry layer takes the bottom-center of her box (her feet), pushes it through this camera's calibrated homography matrix, and out comes a floor-plan coordinate: corridor N, 42.3m east. All of this is stamped with a timestamp from the building's shared clock and published as a tiny JSON event. Centrally, the cross-camera engine notices that track 12 vanished from the north corridor's east edge 9 seconds ago, and a new track just appeared on camera 7 — which the topology graph says is 8–15 walking-seconds away. Physics says plausible; her signature matches at cosine similarity 0.83; the engine assigns both tracklets the same global ID #47. On the guard's dashboard, dot #47 glides from one corridor into the next, one continuous person. That is the whole product.

Step 03 · Orientation · Vol 3 · Mechanics

The whole stack as one data structure

Before the math, hold the shape. If you can name what type of object flows between each stage, you already understand the system better than most people who've read a paper on it. Everything downstream is a transformation of one growing record:

# a pixel grid becomes… (detector) frame : uint8[1080][1920][3] # 6.2M numbers # …a list of boxes… (detector head + NMS) detections : [{box:[x1,y1,x2,y2], score:0.91, cls:0}] # …with stable IDs… (tracker: Kalman + Hungarian) tracks : [{id:12, box, vel:[dx,dy], age, hits}] # …carrying appearance… (Re-ID embedder) tracklet : {cam:3, id:12, signature:float[512], quality:0.8, path:[(x,y,t)...]} # signature = the "fingerprint" # …placed on the floor… (homography) tracklet.path : [(floor_x, floor_y, t) ...] # meters, shared map # …and fused into one identity across cameras (clustering) global_id : {gid:47, members:[(cam3,id12),(cam7,id5)], signature}

Seven arrows. Each section below is one arrow, opened up. The single most important sentence in this volume: the atomic unit of a real system is not the frame and not the detection — it is the tracklet with an aggregated signature. Every robust design is built on that object. A team that talks in frames is a beginner; a team that talks in tracklets has built one.

From the repo · this record is already half-built

Mantis's manifest_schema.md is exactly this data structure minus the identity fields — per frame it carries box_px, class, score, and a world coordinate. You are adding three fields to a record the repo already defines: track_id (Section 05), signature (Section 06), global_id (Section 08). Knowing precisely which fields are new is knowing precisely what you're building.

Step 04 · packages/mantis_data · Vol 1 · Mantis, Decoded

The data engine — half of all CV is this

Nobody puts dataset plumbing in a tutorial, yet it decides more of your final accuracy than the model does. Mantis ingests nine public datasets — drone traffic (VisDrone, UAVDT, AU-AIR), overhead vehicles (VEDAI), satellite scenes (DIOR, DOTA), maritime (SeaShips, SMD, SeaDronesSee) — each with a completely incompatible label format: CSV rows, Pascal VOC XML, one giant JSON, MATLAB files, 8-coordinate rotated boxes. The architecture that tames them is worth memorizing.

The adapter + taxonomy pattern

Every dataset gets an adapter (~150 lines) that knows only its own format quirks. Every decision — which native class maps to which unified class — lives in one declarative YAML file, configs/classes_unified.yaml, which flattens everything into 9 classes (person is id 0; then car, truck, bus, two_wheeler, small_boat, cargo_vessel, passenger_vessel, aircraft). VisDrone's "pedestrian" and "people", SeaDronesSee's "swimmer" — all fold into person. A native class with no mapping is silently dropped. The file's header even forbids putting mapping decisions in adapter code.

Lesson · label-space harmonization

When you combine datasets (and you will — CrowdHuman + MOT20 + your own mall footage), the merge decisions are policy, not code. Keeping them in one reviewable YAML means you can audit "does a child on a scooter count as person?" without reading Python. Copy this pattern verbatim.

The output format is COCO JSON — the lingua franca of detection: an images list, an annotations list with bbox = [x, y, width, height] in pixels, and a categories list. Mantis extends it with provenance keys (source, source_id, original class name, license, commercial_use) that standard tools ignore — lineage rides inside the file forever. One subtle move: every per-dataset file always declares all 9 categories even if it only populates two, so category ids can never drift between files when merging.

Deduplication — why video makes dangerous training data

Three of the datasets are videos sampled into frames. Consecutive frames are near-identical. If you shuffle and split them randomly, near-copies of the same scene land in both train and eval — the model gets graded on data it memorized, and your accuracy numbers become fiction. This is called leakage, and it is the single most common way beginners fool themselves.

The defense here is a perceptual hash (dedup.py): shrink every image to 9×8 grayscale, compare each pixel to its right neighbor, produce a 64-bit fingerprint of the image's gradient structure. Two frames whose fingerprints differ by ≤5 bits (Hamming distance) are declared duplicates and one is dropped — before splitting. It's invariant to re-encoding, resizing, and brightness shifts: exactly the differences between "the same frame stored twice."

Verified gap · residual temporal leakage

The split (union.py) is per-image, stratified per-source. Frames from the same video sequence that differ by more than 5 hash bits (camera panned, cars moved) survive dedup and can still straddle the train/eval boundary. For detection this inflates scores mildly. For your product it's fatal: tracking and Re-ID data must be split per-sequence — the same person seconds apart must never appear in both train and eval. Demand this in every dataset spec you sign off on.

Two more moves worth knowing

  • Stratified split with a floor. Each source contributes 10% of its own images to eval, minimum 1 — otherwise UAVDT's ~80k frames would drown VEDAI's 1,250 and you'd lose all evaluation signal for small domains. Seed is fixed at 42; identical inputs give byte-identical splits. Reproducibility is a feature you specify, not luck.
  • Tiling giant images. DOTA scenes reach ~13,000px. Downscaling would erase the tiny vehicles that are the whole point, so the adapter cuts 1024×1024 tiles with 200px overlap; a ground-truth box joins a tile only if ≥50% of its area is inside. This is training-time SAHI — the same slice-and-merge idea your guide describes for inference.

Also verified: the adapters uniformly discard occlusion/truncation flags the datasets provide, keep zero pure-background images (a known cause of false positives), and never apply UAVDT's "ignore regions" — so unlabeled cars there teach the detector that visible cars are background. Each of these is a deliberate-or-lazy trade you now know to ask about.

The self-bootstrapping trick — your tracker manufactures your Re-ID data

Re-ID training needs pairs of crops known to be the same person. Once your single-camera tracker works (Section 04), it produces those for free: every confident track is one person photographed dozens of times across seconds — different poses, different distances, same identity. Harvest crops from long, clean tracks (drop occluded/tiny ones), treat each track as one identity, and you have a same-camera Re-ID training set from your own space with zero manual labeling. Cross-camera pairs still need a human to confirm — but now they're confirming candidates the system proposes, not labeling from scratch. This loop — tracker output becomes Re-ID training fuel becomes better cross-camera matching — is how the system improves itself after deployment.

02.3 Your own footage — the recording & labeling protocol

Public data gets you to a working prototype. Your data gets you to a working product. The protocol:

  1. Record deliberately, not randomly. From each camera: a quiet period, a busy period, and the hard cases — people crossing paths, sitting down, groups, the handoff zones between cameras. 10–15 minutes per camera per condition beats hours of empty corridor.
  2. Label with CVAT (open source, industry standard). For detection: draw person boxes on every ~10th frame; its interpolation fills between. Budget intuition: ~2,000–5,000 labeled boxes per camera angle is a meaningful fine-tune; a focused evening per camera.
  3. Decide the box convention now: full-extent boxes (estimate the whole person even when half-hidden), not visible-fragment boxes — trackers cope far better when the box keeps meaning "the whole person." Write it in your labeling guide; inconsistency here silently poisons tracking. (Mantis's adapters discarded every occlusion flag — you, indoors, cannot afford to.)
  4. Split per-sequence, never per-frame. Frames from one recording session stay together on one side of the train/eval split. This is the leakage lesson from the mantis review, now load-bearing: violate it and every number you compute lies to you.
  5. Build the golden eval clips (Section 09): 2–3 fully-labeled multi-camera clips, ~5 minutes each, where a human has tracked every person across every camera with consistent global IDs. Laborious (~a day of labeling per clip) and worth more than any benchmark — this is the yardstick every threshold in Sections 04–07 gets tuned against.

02.4 Provenance — do what mantis said, not what it did

From the repo · the pattern, completed

Reuse the whole mantis_data machinery: adapters per dataset, the declarative class-mapping YAML, provenance stamped into every COCO file, dhash dedup, commercial_use most-restrictive-wins at merge. Then add the piece mantis documented but never wrote: the trainer refuses to start unless the dataset manifest's combined license status matches the run's declared tag (research_full vs commercial_clean). It's ~20 lines in the training entrypoint. Every model file you ever produce should carry its data lineage in its MLflow record — under GDPR, "what was this trained on?" is a question a regulator can ask you.

Convolution — seeing, computed by hand

A neural network "sees" by sliding tiny grids of numbers over the image and multiplying. That's the whole trick. Let me prove it moves. Here's a patch of image where the left half is dark (value 10) and the right half bright (90) — a vertical edge:

Worked · multiply matching cells, sum them

That +320 vs 0 is feature detection. The filter "fires" on a dark→light vertical transition and stays silent on flatness. Flip the filter 90° and it detects horizontal edges instead.

Why this becomes "understanding"

One filter finds edges. A convolutional layer is ~64 filters, each finding a different primitive (edges at angles, color blobs). Stack layers, and the second layer's filters slide over the first layer's outputs — so they detect patterns-of-edges: corners, curves, textures. Layer three detects patterns-of-those: a wheel, an eye, a shoulder. By layer forty, a filter fires on "standing human." Nobody designed these filters — they're the ~9 million numbers that training adjusts. That's the one intuition under all of computer vision: simple features composing into complex ones, learned from data.

Judge's tell

Ask "what does the backbone actually do?" A weak answer names architectures ("it's a ResNet"). A strong answer describes the hierarchy: early layers = edges/colors, deep layers = whole objects, all learned. If they can't explain why a network needs depth, they've used a detector but don't understand one.

Step 07 · The detector, mechanism 2 of 2 · Vol 3 · Mechanics

The detector head — from grid to boxes

The backbone turns the image into a stack of feature maps. The head turns those into actual boxes. Here's the machinery most people hand-wave — made concrete.

Picture the image overlaid with a grid, say 80×80 cells (that's what a 640px image at stride 8 gives you). For every one of those 6,400 cells, the anchor-free head outputs a small vector of numbers:

per grid cell, the head predicts: dx, dy # box center offset from this cell w, h # box width & height obj # "objectness": is ANYTHING here? (0–1) p_person # class scores (here just 1 class)

So a raw forward pass emits 6,400 candidate boxes. Almost all are garbage (empty floor, wall) with obj near 0. Two numbers filter them:

Worked · scoring one candidate
cell (40,55) predicts: obj=0.85, p_person=0.92 final score = obj × p_person = 0.85 × 0.92 = 0.78 keep it? score 0.78 ≥ conf_threshold 0.30 → YES, keep a wall cell: obj=0.02 → score ≈ 0.02 → discard

This obj × class factoring is deliberate and clever: it separates "is there something here?" from "what is it?". A blurry half-person can score high on objectness even when the class is uncertain — which is exactly the low-confidence box ByteTrack later rescues (Section 05). Keep that thread in mind.

After thresholding you might still have 30 boxes for 8 real people, because several neighboring cells all fire on the same person. That's what NMS cleans up — next section. The anchor-free label just means the head predicts w,h directly, rather than nudging pre-set template rectangles ("anchors"); one fewer thing to hand-tune, which is why modern YOLO and RT-DETR went anchor-free.

From the repo · you can read this happening

inference.py line-for-line: the model emits [N, 5+num_classes] rows of exactly (cx,cy,w,h,obj, class scores…); the code computes score = obj × best_class, thresholds, then converts cxcywh→xyxy. What you just worked by hand is the actual tensor operation in that file. Reading it after this section, it will read like plain English.

Step 08 · The cleanup step · Vol 3 · Mechanics

IoU & NMS — the overlap arithmetic

IoU — Intersection over Union — is the single most-used number in all of detection and tracking. It measures how much two boxes overlap, 0 (disjoint) to 1 (identical). Worked, because you'll compute it in your head all weekend:

Worked · IoU of two boxes
box A = (100,100)→(200,200) area = 100×100 = 10,000 box B = (150,150)→(250,250) area = 100×100 = 10,000 # intersection = the overlapping rectangle overlap x: 150→200 = 50 wide overlap y: 150→200 = 50 tall intersection = 50×50 = 2,500 union = areaA + areaB − intersection = 10,000 + 10,000 − 2,500 = 17,500 IoU = 2,500 / 17,500 = 0.143 ← slight overlap

Non-Maximum Suppression uses IoU to kill duplicate boxes on one object: sort candidates by score; take the highest; delete every remaining box whose IoU with it exceeds a threshold (say 0.7); repeat. That collapses 30 boxes to 8.

The crowd trap — memorize this, it wins arguments

Two different people standing shoulder-to-shoulder can produce boxes with IoU 0.75. Set the NMS threshold too low (aggressive, say 0.5) and NMS deletes one real person, reading them as a duplicate. This is the mechanism behind "works in the demo, fails in the Saturday food court." The fix indoors: raise NMS IoU to ~0.7–0.75 so only near-identical boxes merge. A judge who asks "how do you handle two people who overlap?" is testing whether you know NMS is a double-edged blade.

Step 09 · packages/mantis_detector · Vol 1 · Mantis, Decoded

The detector — YOLOX-S, end to end

The model is YOLOX-S — an anchor-free, one-stage detector from the YOLO family — consumed through the Apache-2.0 pixeltable-yolox library. "S" means small: depth multiplier 0.33, width 0.50. The repo's own code is deliberately thin: a config, a dataset bridge, MLflow wiring, an evaluator, an exporter. The network, losses, and training loop come from the library untouched. That's the professional default — you don't write detectors, you configure them.

Why YOLOX and not the famous Ultralytics YOLOv8/v11? Licensing. Ultralytics is AGPL-3.0 — importing it into a proprietary product legally obliges you to open-source the product. LICENSES.md names this trap explicitly and even warns that Ultralytics' dataset YAML files carry AGPL headers. This single fact will matter to your mall product on day one, because every tutorial you'll find uses Ultralytics.

The four decisions in the config

config.py changes exactly four things from stock YOLOX-S, and each is a lesson:

  1. 9 classes — the head's classification branch matches the taxonomy.
  2. 1280×1280 input instead of the 640 default. Aerial targets are tiny; resolution is the cheapest small-object fix (your guide's Part 2). Cost: ~4× compute, absorbed by dropping batch size to 8. Indoors, people are large — you'll run 640–960 and bank that compute for more cameras.
  3. Eval every 50 epochs instead of 10 — because in-training multi-GPU evaluation crashed an 8×A100 run twice (an NCCL timeout, dated in a comment). Reading war stories in comments is how you learn what actually breaks.
  4. Everything else inherited — 300 epochs, mosaic + mixup augmentation, cosine LR, EMA — with a comment: "intentionally untouched so this is a baseline." Establish a clean reference before tuning. Always.
Lesson · the augmentations that matter

Mosaic stitches 4 random training images into one, jittering scale 0.1–2×; mixup blends image pairs. Both force the model to see objects at unusual scales and contexts — powerful for small objects, and standard in every modern YOLO recipe. The last 15 epochs run augmentation-free so the model finishes on clean, realistic imagery. For your mall model, the analog is random erasing (from your deep-dive guide) — synthetic occlusion, because occlusion is the indoor enemy.

One genuinely surprising verified fact: training starts from random weights. There is no COCO-pretrained checkpoint anywhere — 300 from-scratch epochs is the plan. Defensible for a specialized 9-class aerial domain; for your people detector, you will do the opposite (start from pretrained weights, fine-tune on people data) and converge in a fraction of the time.

Inference: the five steps every frame takes

inference.py is the cleanest 200-line summary of real detector plumbing you'll find:

  1. Letterbox — resize preserving aspect ratio into a 1280×1280 canvas, pad the remainder with gray (value 114), keep the scale ratio. YOLOX quirk: image anchored top-left, pixels stay BGR 0–255, no normalization.
  2. Forward pass — the model emits ~thousands of candidate rows: center-x, center-y, w, h, objectness, 9 class scores.
  3. Score & threshold — score = objectness × best class score; drop below conf (0.30 in the tools).
  4. Un-letterbox — divide coordinates by the saved ratio to get original-frame pixels.
  5. Class-aware NMS at IoU 0.65 — duplicate boxes of the same class merge; a person box never suppresses an overlapping car box.

Those two numbers — conf and NMS IoU — are your guide's "two knobs you tune forever," and the repo quietly proves the point: the CLI default is 0.30, but the Studio web form ships 0.6. Someone tuned the demo for a clean map over recall and never reconciled the defaults. You've now caught your first real config drift.

Evaluation and export

Eval uses pycocotools COCO mAP (AP@50:95, AP50, AP75) — but the smart part is per-source scoring: the same detections re-scored per dataset domain, because an aggregate number hides "great on land, blind at sea." At eval time the confidence threshold drops to 0.01 — mAP integrates the whole precision-recall curve, so eval wants everything; only display wants 0.30+. Export goes to ONNX (portable model graph) with a mandatory parity check: run the same input through PyTorch and ONNX Runtime, require answers to match within tolerance, exit nonzero otherwise. No TensorRT, no INT8 quantization anywhere — the deployment story stops at ONNX. Your guide's Part 6 optimization chain remains yours to build.

Step 10 · Model 1 · Vol 2 · Builder’s Manual

The person detector

Job: one frame in → a list of (box, "person", confidence) out. No memory, no identity — it meets every frame as a stranger. Everything downstream depends on its quality, and no tracker can recover a person the detector never found, so recall is king: a missed person is worse than an occasional ghost (the tracker filters ghosts).

What's actually inside

YOLOX-S, like every modern detector, is three stages of learned convolutions: a backbone that turns pixels into feature maps (early layers detect edges, middle layers body parts, deep layers whole-person patterns), a neck that fuses coarse-but-smart deep features with fine-but-naive shallow ones (so it can find both near and far people), and a head that reads the fused features and emits, for each location on a grid, box coordinates + an objectness score + class scores. It's anchor-free: it predicts box centers and sizes directly. At inference your five steps are letterbox → forward pass → score = objectness × class → un-letterbox → NMS. You watched all five in the repo; they carry over byte-for-byte.

What fine-tuning actually does

Mechanically: each training step takes a batch of labeled images, runs the forward pass, and a loss function scores the wrongness — three terms added together: box-overlap loss (predicted boxes should align with true boxes, measured by IoU), objectness loss (grid cells with a person should say so), classification loss. Backpropagation computes, for all ~9 million weights, which direction reduces that loss, and nudges each a tiny step (the learning rate). Repeat over the dataset ~40 times (epochs). Fine-tuning is the same loop starting from pretrained weights with a ~10× smaller learning rate — you're adjusting a competent model to your domain, not teaching it vision. What you'll actually watch: the loss curve falling (MLflow plots it), and eval mAP on your held-out camera clips rising then plateauing — stop there; past the plateau it starts memorizing your training set (overfitting), which shows as train loss still falling while eval mAP degrades.

Indoor-specific traps to engineer for

  • The mannequin/poster/mirror trio — the classic mall false positives. Fix per camera with ignore masks: polygons over ad screens and mirrors where detections are discarded. (Mantis skipped applying ignore regions; you shouldn't.)
  • Zero negatives problem — mantis's corpus contains no empty scenes, which teaches false confidence. Include empty-corridor frames from every camera in training.
  • Evaluate per camera — reuse the per-source eval trick (eval.py): tag every image with its camera ID and slice mAP by camera. The dim car-park camera will be your weakest — you want to know that from a table, not a complaint.

Definition of done: person mAP@50 ≥ ~0.85 on your own labeled eval clips, per camera, at your chosen operating resolution — plus a subjective test: at conf 0.3, watching live output, it should almost never miss a clearly visible person for more than a frame or two.

Step 11 · The tracker, mechanism 1 of 2 · Vol 3 · Mechanics

The Kalman filter — a real update

The Kalman filter is described as intimidating and is actually two steps: guess where they'll be, then blend your guess with what you measured. It's what lets a track coast through occlusion and skipped frames. Let me run one full cycle with numbers.

Track 12's state is position and velocity: she's at pixel x=100, moving +30 px/frame. Each number has an uncertainty (variance). Here's one frame:

Worked · predict, then correct
# STEP 1 — PREDICT (apply momentum, uncertainty grows) new pos = old pos + velocity = 100 + 30 = 130 uncertainty grows: P: 10 → 21 # less sure after coasting # STEP 2 — a detection arrives at pixel 127 (noisy, we trust it ±5) innovation = measured − predicted = 127 − 130 = −3 Kalman gain K = 0.457 # how much to trust the measurement updated pos = 130 + K×(−3) = 130 − 1.37 = 128.6 updated vel = 30 + 0.217×(−3) = 29.3 uncertainty shrinks: 21 → 11 # more sure after seeing evidence

Read what the gain did: the prediction said 130, the measurement said 127, and the filter landed at 128.6 — between them, closer to the prediction because this frame it happened to trust the model slightly more. If the detector were noisier (trust ±20 instead of ±5), the gain would shrink and it would lean on the prediction even harder. That self-balancing between "what physics says" and "what I just saw" is the entire filter.

Why this is the thing that makes tracking work

Now the payoff. A person walks behind a pillar for 8 frames — no detections at all. The filter just runs STEP 1 repeatedly: 130, 160, 190… uncertainty growing each time. When she emerges and a detection appears near the predicted 250, it matches the coasting track and reclaims ID 12. Without the Kalman filter, every occlusion births a new ID (fragmentation). With it, the track survives the gap. This is also precisely why you can sample at 5fps instead of 30 and still get smooth tracks — the filter fills the 4 unseen frames between each detection.

Judge's tell

"What happens when someone walks behind a pillar?" Weak: "we lose them / start a new track." Strong: "the motion model coasts the track for up to track_buffer frames and re-acquires it on the other side; if the occlusion outlasts the buffer we fragment, which is why we tuned the buffer to our camera's typical occlusion length." The second answer means they've actually watched their own failure modes.

Greedy grabbed the tempting 1.0 and got trapped into a catastrophic 9.0 — an ID switch, the cardinal tracking sin. The Hungarian algorithm considers all pairings jointly and finds 3.1. That 6.9-cost difference is a person's identity being swapped versus preserved. Every serious tracker (SORT, DeepSORT, ByteTrack) solves the assignment, not the greedy grab.

ByteTrack's actual innovation, precisely

Now the piece that made ByteTrack the default. Standard trackers throw away low-confidence detections (score < 0.5) as noise. ByteTrack keeps them for a second assignment round:

# ROUND 1: match tracks to HIGH-confidence detections (score > 0.5) → most people, matched cleanly # ROUND 2: take the STILL-UNMATCHED tracks, and match them # against the LOW-confidence leftovers (score 0.1–0.5) → the half-occluded woman whose box dropped to 0.3 gets matched instead of lost ← the whole trick

Why it works: when someone is partly hidden, the detector still fires but with low confidence. That weak box is not noise — it's a real person the detector is unsure about. Rescuing it in round 2 is why ByteTrack holds identities through crowds where older trackers shatter. This is also why the detector's confidence threshold and the tracker's thresholds must be tuned together — set detector conf too high and you throw away the boxes round 2 needs.

The tracker — boxes become people

Job: consume each frame's anonymous boxes and link them over time, so "a person was here" becomes "this person, track 12, has been walking east for 40 seconds." It's not a neural network — it's an algorithm (with a tiny bit of learned help later), which is why you can tune it with thresholds instead of retraining.

The machinery, concretely

Use ByteTrack — the modern default, and pick an MIT-licensed implementation (the original ByteTrack repo, or the supervision library's port — not boxmot, which is AGPL like Ultralytics). Every processed frame, three things happen:

  1. Predict. Each existing track has a Kalman filter — a running estimate of position, box size, and velocity, plus an uncertainty. It applies momentum: "track 12 was here moving 30px/frame east, so she should now be there." Uncertainty grows the longer since the last confirmed sighting. This is what lets a track coast through a pillar occlusion or the frames you skipped by sampling.
  2. Match. Build a cost matrix: every predicted track position × every fresh detection, scored by overlap (IoU). The Hungarian algorithm solves the best one-to-one assignment. ByteTrack's signature move is doing this twice: first pass matches confident detections (score > ~0.5); the second pass matches the leftovers against low-confidence detections (score down to ~0.1) — because a half-occluded person's confidence drops but the weak box still exists, and that weak box is what keeps her track alive through the crowd. This one idea is most of ByteTrack's advantage, and it's why detector and tracker thresholds must be tuned together.
  3. Lifecycle. Unmatched detections become tentative tracks (confirmed only after ~3 consecutive hits — kills one-frame ghosts). Unmatched tracks go lost and coast for track_buffer frames before deletion; a reappearing detection within the buffer reclaims its old ID.

PART V

Re-identification

Appearance as a vector: the math that moves, then the embedder you deploy and tune.

Step 14 · Re-ID, the heart · Vol 3 · Mechanics

Embeddings & triplet loss — a step that moves

An embedding turns a person-crop into a list of numbers (say 512 of them) positioned so that same-person crops land near each other and different-person crops land far apart. "Near" is cosine similarity. Worked, small, so the mechanism is naked:

Worked · cosine similarity of two signatures
signature u = [1, 2, 2, 0] # 4-dim toy (real = 512) signature v = [2, 3, 1, 1] dot(u,v) = 1·2 + 2·3 + 2·1 + 0·1 = 10 |u| = √(1+4+4+0) = 3.00 |v| = √(4+9+1+1) = 3.87 cos = 10 / (3.00 × 3.87) = 0.861 ← quite similar # threshold decides: 0.861 ≥ 0.7? → SAME person

Cosine near 1 = same person; near 0 = unrelated. That one threshold is your product's master dial (Volume 2, §05). After normalizing the vectors, cosine and Euclidean distance rank pairs identically — which is why papers switch between them freely and mean the same thing.

How the embedder learns that geometry — triplet loss, moving

How do you train a network so same-person lands close? Show it triples and punish it when the geometry is wrong. A triple: anchor (a crop of person X), positive (another crop of X), negative (a crop of person Y).

Worked · one triplet loss step
current distances (want anchor CLOSE to positive, FAR from negative): d(anchor, positive) = 0.40 # same person — too far! d(anchor, negative) = 0.55 # different person — too close! margin m = 0.30 loss = max(0, d(a,p) − d(a,n) + m) = max(0, 0.40 − 0.55 + 0.30) = max(0, 0.15) = 0.15 ← POSITIVE, so we learn # gradient descent now nudges weights to: # pull anchor & positive together (shrink 0.40) # push anchor & negative apart (grow 0.55) # until d(a,n) > d(a,p) + 0.30 → loss hits 0, nothing to learn

Repeat over millions of triples and the 512-dim space self-organizes by identity. The margin (0.30) is the enforced safety gap — same-person must be closer than different-person by at least that much.

Where it fails — and why you must know this cold

Batch-hard mining: random triples are mostly easy (loss already 0, nothing learned), so training deliberately picks the hardest ones — the same-person pair that looks most different, the different-person pair that looks most alike. Consequence you can state to any judge: the model can only tell apart what its training data forced it to. Twenty people in identical black hoodies were never a hard case in Market-1501, so the embedder will collapse them to nearly one point — and no threshold fixes that. This is exactly why the next section's physics gates exist: appearance alone drowns in a uniform crowd.

Step 15 · Model 3 · Vol 2 · Builder’s Manual

The Re-ID embedder — appearance as a number

Job: person crop in → vector of 512 numbers out, engineered so that two crops of the same person land close together and crops of different people land far apart — across cameras, angles, and lighting. It is not face recognition; it never uses the face (CCTV faces are too few pixels anyway). It keys on whole-body gestalt: clothing colors, silhouette, bag, hair. Closeness is cosine similarity — near 1.0 = almost certainly the same person, and cross-camera matching becomes nearest-neighbor search over these vectors. This anonymous-appearance choice is also your legal keel: session-scoped body matching sits in a workable GDPR/AI-Act position; face-ID against named databases does not (your overview guide, Part 9).

Architecture and training, at the depth you need

Use OSNet (via the MIT-licensed torchreid library) — designed specifically for Re-ID, and cheap enough to run on live streams. Its idea: identity lives at many scales at once (overall silhouette and a small logo and shoe color), so each block processes several receptive-field sizes in parallel and learns to weight them. How such a model is trained tells you where it fails: triplet loss shows it an anchor crop, a positive (same person), and a negative (different person), and pushes the anchor closer to the positive than the negative by a margin — millions of times, with batch-hard mining deliberately selecting the most confusable cases, plus random erasing (blanking patches to simulate occlusion). Consequence: the model can only separate what its training data forced it to separate. Twenty teenagers in identical black hoodies were never in Market-1501 — your mall will provide them. That is why Section 07 refuses to let appearance work alone, and why fine-tuning on your own cameras (with the self-harvested pairs from Section 02) is not optional for production quality.

Provenance warning · pretrained Re-ID weights are research-tainted

Every downloadable OSNet checkpoint was trained on Market-1501/MSMT17 — research-only data. Fine for your prototype; not shippable. Your clean path for the product: pretrain on synthetic identities you generate + fine-tune on your own consented footage, tracked by the provenance tags of Section 02.4. Budget for this now so it isn't a crisis later.

From frames to signatures — the tracklet is the atom

Never match on a single frame's embedding — any one frame can be blurred, mid-stride, half-occluded. Instead maintain a track-level template: for each healthy track, embed only the decent crops (gates: box height ≥ ~80px, not flagged occluded, not at frame edge) and fold them into a running average — an EMA, template ← 0.9·template + 0.1·new — which stays stable while adapting as the person turns or walks into shade. The unit that flows to Section 07 is a tracklet: (camera, track ID, time span, floor-plan path, aggregated signature, quality score). A tracklet built from 60 clean frames deserves trust; one built from 2 blurry frames doesn't — carry that quality number and use it.

The threshold — your product's central dial

One number decides whether two tracklets are "the same person": the cosine-similarity cutoff. Too high → the same person under new lighting fails to match → fragmentation (one human becomes five dots; phantom crowds). Too low → look-alikes merge → collision (the guard follows dot #47 and it's sometimes a stranger — silent and dangerous). There is no universal value. The tuning procedure: from your golden clips, collect a few hundred labeled tracklet pairs (same/different), plot match-rate vs error-rate as you sweep the threshold, and pick the operating point — per camera pair, because the dim corridor→lift handoff needs a different cutoff than two bright adjacent cameras. Expect values in the 0.5–0.7 range with OSNet; expect to re-tune after any embedder change.

Step 16 · The geometry · Vol 3 · Mechanics

Homography — one matrix multiply

A homography is a 3×3 matrix that maps any point on the camera's floor to a point on your floor-plan — exactly, for a flat floor, forever, once calibrated. Here's the whole operation: take a foot-pixel, make it a 3-vector by appending 1, multiply, divide out the third number.

Worked · project a foot pixel to the floor plan
H = | 0.80 0.10 50 | foot pixel = (600, 400) | 0.00 1.20 30 | as a vector: (600, 400, 1) |0.0005 0.0002 1 | # multiply H × (600,400,1): x' = 0.80·600 + 0.10·400 + 50·1 = 570 y' = 0.00·600 + 1.20·400 + 30·1 = 510 w' = 0.0005·600 + 0.0002·400 + 1 = 1.38 # divide by w' — this is the "perspective" step: floor_x = 570 / 1.38 = 413 floor_y = 510 / 1.38 = 370 → person is at floor-plan (413, 370) # centimeters, say

That division by w' is what makes it perspective rather than a flat stretch — points farther from the camera get compressed exactly the way a receding floor looks. The 8 free numbers in H (the 9th is fixed by that division) are why you need 4 point-correspondences to solve for it: each gives 2 equations, 4×2 = 8.

You never compute H by hand — cv2.findHomography takes your clicked point-pairs and returns it, using RANSAC to ignore your one fat-fingered click (it fits many candidate matrices from random 4-point subsets and keeps the one the most points agree with). But now you know what the returned matrix does to a pixel.

The two failure modes to name

(1) Wrong pixel: the homography is only exact for points on the floor, so you project the box's bottom-center (the feet). Project the box center and you place people a meter deep into the floor. Occluded feet (behind a counter) push the point too far — the same error terrain causes for the drone. (2) Wrong plane: "flat floor" breaks on stairs, escalators, mezzanines — each needs its own homography. A judge asking "what about the escalator?" is checking if you know a single homography assumes a single plane.

Step 17 · geolocate.py + dji.py · Vol 1 · Mantis, Decoded

Geometry — pixels onto the world

This is mantis's answer to the question your whole product hinges on: a detector found something at pixel (px, py) — where is it in the world? The drone answer, 77 lines in geolocate.py, verified line by line:

  1. Meters-per-pixel at the aim point: GSD = 2 · slant_range · tan(HFOV/2) ÷ image_width. This one formula is the camera intrinsics — focal length derived from field of view, assuming a centered, undistorted lens.
  2. Look-down angle (depression) = asin(altitude ÷ slant_range) — pure flat-ground trigonometry.
  3. Pixel offsets from image center: sideways offset = du × GSD; forward offset = −dv × GSD ÷ sin(depression). That division is foreshortening: the shallower you look, the more ground each image row covers. A floor of sin ≥ 0.15 stops near-horizon views from projecting to infinity.
  4. Rotate by look azimuth into north/east meters, convert to degrees (111,320 m per degree of latitude, cos-scaled longitude). Done.

The crucial subtlety, confirmed in source: this is not a true ray-cast and not a homography. It's a linearization — exact at the image center (anchored to the telemetry's reported frame-center coordinate), first-order approximate everywhere else, with error growing toward the frame edges. Camera roll is ignored entirely (DJI gimbals stabilize it to ~0). The docstring is honest: "real-ish, not survey-grade: ignores Earth curvature, terrain, and lens distortion."

Step 18 · The spatial substrate · Vol 2 · Builder’s Manual

The geometry layer — every camera onto one map

Job: convert "pixel (612, 480) in camera 3" into "floor-plan point (42.3m, 17.8m)". Because your cameras are fixed and your floor is flat, this is exact math done once, not a learned model — the one part of the system that either works or has a diagnosable error.

Calibrating one camera — the actual procedure

  1. Undistort first. Wide-angle CCTV lenses bend straight lines. Print a checkerboard, wave it through ~20 frames, run OpenCV's calibration to get the distortion model, and undistort every frame from then on. Skipping this makes step 3 accurate in the center and wrong at the edges — exactly where handoffs happen. (Fisheye/dome cameras need the dedicated fisheye model — flag them early.)
  2. Collect correspondences. Find ≥8 points visible in the camera whose floor-plan positions you know: tile corners (a tiled floor is a free calibration grid), doorway edges, pillar bases. Click each in the camera image; click its twin on the floor-plan PNG.
  3. Fit the homography — cv2.findHomography(..., RANSAC) returns the 3×3 matrix mapping any floor pixel to floor-plan coordinates. RANSAC discards your one fat-fingered click. Store the matrix; it's valid until someone physically moves the camera.
  4. Verify by round trip. Project 3–4 held-out floor marks through the matrix; error should be well under 0.5m. Then automate it: re-check nightly and alert on drift — a free "did a ladder bump camera 6?" alarm. (This is mantis's forward/inverse round-trip test pattern, redeployed.)

At runtime, project each person's foot point — bottom-center of the box — because homographies are only exact for points on the floor plane. Occluded feet (a counter, another person) put the mapped point too far away; the upgrade path when this bites is ankle keypoints from a pose model. And the mantis error-budget lesson holds: the guard needs "which corridor," so a meter of accuracy is success — spend effort on foot points and calibration, not sub-pixel heroics.

Two structures beyond single-camera calibration

  • Overlap fusion. Where two cameras see the same floor, the same person projects to (nearly) the same map point from both — fuse points within a tolerance radius (~0.5–1m; tune on overlap footage). In overlap zones geometry leads and appearance rests: it's the reliable regime.   • The topology graph. For non-overlapping pairs: which camera exits connect to which entries, and how long the walk takes. Later, learn it from data: log actual gaps of high-confidence matches and let the distributions sharpen themselves. Step 19 · packages/mantis_reencoder · Vol 1 · Mantis, Decoded

Time & telemetry — the hidden backbone

This is the subsystem with the least YouTube-tutorial glamour and the most transferable wisdom. A drone's video is useless for mapping without knowing, per frame, where the camera was and where it pointed. Militaries standardized that: KLV (key-length-value binary packets) per MISB ST 0601 (which fields mean what), muxed into the video file per STANAG-4609. Mantis implements a 26-tag encoder/decoder from scratch: platform lat/lon/altitude, heading, gimbal angles, field of view, slant range, frame center — each value squeezed into 2–4 bytes by mapping its physical range onto an integer range (latitude resolution works out to ~5mm).

You will never use KLV in a mall. You will use its lessons daily:

Lesson 1 · streams are joined by time, never by index

Telemetry arrives asynchronously, at a different rate than video. So every packet carries its own microsecond timestamp (Tag 2), every video frame gets a computed timeline position, and the join is nearest-timestamp within a tolerance (±2 frame periods here) via binary search. Outside tolerance → the frame honestly reports "no pose" instead of using a stale one. Your mall system is nothing but this problem multiplied: 20 cameras, each with its own clock, fused by time. Clock sync (NTP/PTP) plus this exact nearest-with-tolerance join is the backbone of cross-camera identity — your deep-dive guide's section C7, here as working code.

Lesson 2 · never trust a codec you only round-tripped

A systematic scaling bug encodes and decodes consistently — your own tests pass while every value is wrong. So test_klv_oracle.py decodes a real 2009 drone sample through an independent third-party parser and requires tag-for-tag agreement. This is oracle testing. Apply it to anything you build that has a spec: validate against an independent implementation and real captured data, not your own mirror image.

Lesson 3 · synthetic data buys you the pipeline before reality arrives

The real test flight looked 0.37° above horizontal — a useless grazing view. Why it's clustering, not lookup — the one conceptual upgrade

The naive design assigns each new tracklet to its nearest existing ID, one at a time (greedy — the same trap as Section 05, now global). The correct design collects all tracklets in a time window, builds a graph where edge weights are these fused scores, and clusters it — partitioning into groups where each group is one human, respecting the hard constraint that one person can't be two places at once (a "cannot-link" between overlapping-in-time tracklets). Solving it jointly means an early ambiguous match can be revised when better evidence arrives, instead of cascading into errors.

Judge's tell — this one question separates the top team

"How do you tell apart two people dressed identically?" The losing answer is about a better embedder ("we use a bigger Re-ID model"). The winning answer is: "appearance can't — we gate on space-time first. The camera topology says only the person who physically could have made that transit is a candidate, so appearance only disambiguates among the few who survive the physics gate." A team that says this has understood the actual hard problem. A team that only talks about their model has not.

Step 21 · Model 4 — the core IP · Vol 2 · Builder’s Manual

The cross-camera engine — one human, one ID

Job: take every camera's tracklets and partition them into groups — one group per real human — assigning each group a global ID. This is the piece that makes twenty screens into one dashboard, the hardest thing you'll build, and the part with no library to install. It's also where beginners make one specific wrong turn, so let's name it.

Not a lookup — a clustering problem under physics

The naive design — "new tracklet appears, find its nearest gallery signature, assign that ID" — is greedy: early mistakes lock in and cascade. The correct frame: over a sliding window (say the last 30–60 seconds), collect all tracklets from all cameras, build a graph where edges score the likelihood two tracklets are the same person, and cluster the graph — merging the closest pair repeatedly, refusing merges that violate constraints — so the whole window resolves jointly. If tracklets P, Q, R all look alike, joint resolution can see that R is Q's better match before greedily marrying P to Q.

The edge score — physics gates, appearance disambiguates

Each candidate edge is scored in a strict order:

  1. Hard gates first (kill switches). Same-camera-same-time → impossible (a person isn't in two places; also the cannot-link constraint inside clustering). No topology path → impossible. Transit time outside the pair's distribution (too fast = teleporting) → impossible. Edge weight zero, regardless of appearance.
  2. Space-time plausibility. Observed gap scored against the transition distribution — 11 seconds on an 8–15s corridor scores high; 40 seconds scores low but nonzero (they may have lingered in the blind gap).
  3. Appearance. Cosine similarity of the tracklet templates (Section 05), thresholded per camera pair.
  4. Motion continuity. Exit velocity should roughly aim at the entry point; exiting left and appearing from the wrong side scores down.
  5. Fuse — a weighted sum works to start (roughly: appearance 50%, space-time 35%, motion 15%; a small learned model later). Merge clusters greedily-highest-first while no constraint breaks; anything unmatched enrolls as a new global ID — the open-world default, since most new tracklets are genuinely new people.

This ordering is what defeats the black-hoodie problem: of twenty identical hoodies, physics admits only the two who were in the right place at the right time — appearance only has to pick between those.

Global ID lifecycle — the state you must manage

The engine holds the live gallery: global ID → member tracklets, current best signature, last-seen time and place. Two dials rule it: expiry (drop IDs unseen for ~5–10 minutes — too short and a returning lingerer becomes a new person; too long and stale signatures hijack newcomers — the track_buffer trade at building scale) and the match threshold (Section 05's fragmentation-vs-collision dial, now global). Then three systems truths from your deep-dive guide, which become real the week you go multi-machine: fuse by timestamps, never arrival order (events arrive late and out of order — buffer a few seconds and reason by time); clock skew silently corrupts every transit-time gate (NTP discipline, Section 01); and log every merge decision with its evidence, because when a guard asks "why did the dot jump?", you want an answer, not a shrug.

From the repo · the join you already understand

The mechanism at this engine's heart — match events across streams by nearest timestamp within a tolerance, degrade honestly when nothing qualifies — is exactly PoseTrack.nearest() from metadata/read.py, generalized. Mantis joined video-to-telemetry; you join camera-to-camera. Same discipline, same failure modes, same fix.

Definition of done: on your golden multi-camera clips, cross-camera IDF1 climbing past ~0.6–0.7, and the qualitative bar: a colleague walks a loop through all cameras and stays one dot ≥8 times out of 10.

Step 24 · CI, licensing, provenance · Vol 1 · Mantis, Decoded

The discipline layer — what separates a repo from a product

Mantis's most sophisticated subsystem contains no computer vision at all. It's the machinery that governs what's allowed to enter the codebase and leave it — and for a company selling proprietary software built on open-source ML, that machinery is existential.

Three layers of license enforcement

  • At commit: a pre-commit hook scans every Python dependency's declared license and refuses GPL/AGPL. You literally cannot commit a forbidden dependency.
  • At merge: CI re-runs the same gate, plus an equivalent gate over all npm packages (which would mechanically reject react-leaflet's "ethical source" license).
  • Weekly: a scheduled scancode job scans the source files themselves for embedded GPL/AGPL headers — catching code copy-pasted from, say, an Ultralytics repo, which no dependency scan can see.

Exceptions require written justification in the code. The policy doc, the scripts, and CI all agree. For your product — where the tempting default stack (Ultralytics) is AGPL and the classic datasets are non-commercial — copy this apparatus wholesale on day one.

Provenance: models inherit their data's law

RESEARCH_ONLY.md encodes an idea most teams learn painfully: weights trained on non-commercial data are non-commercial artifacts. All nine mantis datasets are commercial-use false or unclear, so Month-1 weights may never ship, may not even be fine-tuned into a product model. The policy defines MLflow tags (research_full / commercial_clean / synthetic_only) and claims the trainer refuses to launch without a consistent tag.

Verified gap · the enforcement doesn't exist

I grepped the entire repo: dataset_set appears only in the policy document. No code sets it, checks it, or blocks training without it. The document even forbids "removing or weakening" an enforcement that was never written. The policy is real; the mechanism is aspirational. Two lessons: (1) when a doc says "the system enforces X," verify in code — you just watched it fail in a professional repo; (2) for your product this is the pattern to implement for real, because your world has the same trap: Market-1501, MSMT17, MOT17/20, CrowdHuman are all research-grade licenses, and under GDPR your training data provenance is a legal question, not just a contractual one.

Smaller habits worth stealing

  • One lockfile, frozen everywhere — dev, CI, and Docker all resolve the identical environment; drift fails the build.
  • Two Dockerfiles — a fat CUDA training image and a slim, non-root runtime image (your training rig vs your per-site edge box), including a documented trap: syncing dependencies inside the GPU container would silently replace GPU PyTorch with the CPU build "and training would never use the card."
  • Strictness where bugs are silent — the type checker is permissive globally but maximally strict on the one package (the binary telemetry codec) where a type confusion corrupts data quietly. In your product, that package is the cross-camera fusion engine.
  • 1MB commit cap — model weights and datasets mechanically cannot enter git; they flow through DVC (data version control). Though verified: the DVC remote is a path in one developer's home directory and no data is actually tracked yet — scaffolding, not practice.

  4. Anonymous by design, and say so. "Session-scoped IDs, no face recognition, no identity database — which is also what keeps it lawful under GDPR and the EU AI Act." Nails the ethics question before it's asked.
  5. Right-size the accuracy claim. "We localize to about a meter, which is all a guard needs — which shop, which corridor." Claiming centimeters would out you as someone who hasn't measured.
Demo-day landmines (these kill more demos than bad code)

Conference WiFi dies → run everything local, no cloud calls, basemap tiles cached (mantis's exact trick: a fallback so the map shows dots on black, not broken tiles). Lighting is different from your practice room → recalibrate homography on-site in the morning, and set detector conf a notch low so you don't miss people in front of the judges. Live inference stutters on the venue laptop → have the backup recording ready. The clock desync bug → NTP all machines the night before. Every one of these has ended a good project.

The BS detector — five ways to catch a bluff in one minute

If you have 60 seconds, ask these
  1. "Show me it failing." A team that has run their own system can produce a failure instantly and explain it. A team that hasn't will stall or claim it doesn't fail — the tell.
  2. "What's your track_buffer / similarity threshold set to, and why that number?" A real builder has a specific number and a reason ("60 frames, ~2s occlusions in our space"). A bluffer gives a vague range or "the defaults."
  3. "Is this live or a recording?" (You saw in the mantis review that a "LIVE" badge can just mean "video playing.") Ask them to walk a new person in front of the camera now. Live systems survive it; demos of a saved clip can't.
  4. "Where does the cross-camera match actually happen — in the browser or on the server?" A browser-side greedy matcher (like mantis's tracks.ts) is a cosmetic tracker, not real MTMC. The answer reveals whether identity is real or decorative.
  5. "What data did you train on, and can you ship a model trained on it?" If they trained on CrowdHuman / Market-1501 and plan to sell it, they've missed the research-only license trap — a tell they haven't thought past the demo.
